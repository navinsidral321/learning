Low-Level Design (LLD)

Low-Level Design focuses on detailed component design, class relationships, and code-level implementation patterns.

1. Class Design & Encapsulation

  • Encapsulation: Bundle data and methods, hide internal implementation from external access using access modifiers
  • Single Responsibility: Each class has ONE reason to change and ONE primary responsibility
  • Composition over Inheritance: Prefer composition to inheritance for flexibility and avoiding rigid hierarchies
  • Clear Public Interfaces: Minimal, purpose-driven public methods; rest should be private
  • Dependency Injection: Pass dependencies explicitly through constructors rather than creating internally

2. Design Patterns in LLD

Creational Patterns:

  • Singleton: One instance only (DatabaseConnection, Logger) - thread-safe using Bill Pugh pattern
  • Factory: Create objects without specifying exact classes (PaymentGatewayFactory)
  • Builder: Construct complex objects step-by-step (OrderBuilder for many parameters)

Structural Patterns:

  • Adapter: Make incompatible interfaces work together (Stripe API to payment interface)
  • Decorator: Add features dynamically without modifying original (CachedPaymentGateway wraps gateway)
  • Facade: Provide simplified interface to complex subsystem (OrderProcessingFacade)

Behavioral Patterns:

  • Observer: Notify multiple objects about state changes (OrderObserver for email, SMS)
  • Strategy: Interchangeable algorithms at runtime (PercentageDiscount vs FlatDiscount)
  • State: Object behavior changes with state (Order: Pending → Confirmed → Shipped)
  • Template Method: Define algorithm skeleton, subclasses override steps
  • Chain of Responsibility: Pass request through handler chain until handled

Strategy Pattern Example:

// Strategy Pattern for flexible discount strategies
public interface DiscountStrategy {
    double applyDiscount(double amount);
}

public class PercentageDiscount implements DiscountStrategy {
    private final double percentage;
    public PercentageDiscount(double percentage) { this.percentage = percentage; }
    
    @Override
    public double applyDiscount(double amount) {
        return amount * (1 - percentage / 100);
    }
}

public class FlatDiscount implements DiscountStrategy {
    private final double flatAmount;
    public FlatDiscount(double flatAmount) { this.flatAmount = flatAmount; }
    
    @Override
    public double applyDiscount(double amount) {
        return Math.max(0, amount - flatAmount);
    }
}

public class Order {
    private double totalAmount;
    private DiscountStrategy discountStrategy;
    
    public void setDiscountStrategy(DiscountStrategy strategy) {
        this.discountStrategy = strategy;
    }
    
    public double getFinalPrice() {
        if (discountStrategy == null) return totalAmount;
        return discountStrategy.applyDiscount(totalAmount);
    }
}

3. State Management & Immutability

  • Immutability: Create immutable objects for thread safety (final fields, no setters)
  • Builder Pattern: For complex objects with many optional parameters
  • State Machines: Model lifecycle with defined states and valid transitions
  • Event Sourcing: Track all changes as immutable events for auditability
  • Validations: Validate state transitions and class invariants

Immutable Order with Builder:

// Immutable Order class
public class Order {
    private final String orderId;
    private final String userId;
    private final List<OrderItem> items;
    private final OrderStatus status;
    private final BigDecimal totalAmount;
    private final LocalDateTime createdAt;
    
    private Order(Builder builder) {
        this.orderId = builder.orderId;
        this.userId = builder.userId;
        this.items = Collections.unmodifiableList(builder.items);
        this.status = builder.status;
        this.totalAmount = builder.totalAmount;
        this.createdAt = builder.createdAt;
    }
    
    public static class Builder {
        private final String orderId;
        private final String userId;
        private List<OrderItem> items = new ArrayList<>();
        private OrderStatus status = OrderStatus.PENDING;
        private BigDecimal totalAmount;
        private LocalDateTime createdAt = LocalDateTime.now();
        
        public Builder(String orderId, String userId) {
            this.orderId = orderId;
            this.userId = userId;
        }
        
        public Builder addItem(OrderItem item) {
            this.items.add(item);
            return this;
        }
        
        public Builder withStatus(OrderStatus status) {
            this.status = status;
            return this;
        }
        
        public Order build() {
            this.totalAmount = items.stream()
                .map(item -> item.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())))
                .reduce(BigDecimal.ZERO, BigDecimal::add);
            
            return new Order(this);
        }
    }
    
    // All getters, no setters - immutable
    public String getOrderId() { return orderId; }
    public String getUserId() { return userId; }
    public List<OrderItem> getItems() { return items; }
}

4. Exception Handling & Error Recovery

  • Custom Exceptions: Create domain-specific exceptions for clarity (PaymentException, ValidationException)
  • Exception Hierarchy: Checked for recoverable errors, unchecked for programming errors
  • Graceful Degradation: Handle failures appropriately without propagating all exceptions
  • Logging with Context: Log exceptions with relevant context for debugging
  • Retry Logic: Implement exponential backoff for transient failures

Exception Hierarchy Example:

// Custom exception hierarchy
public abstract class DomainException extends RuntimeException {
    protected DomainException(String message) { super(message); }
    protected DomainException(String message, Throwable cause) { super(message, cause); }
}

public class PaymentException extends DomainException {
    private final PaymentErrorCode errorCode;
    public PaymentException(String message, PaymentErrorCode code) {
        super(message);
        this.errorCode = code;
    }
}

// Retry with exponential backoff
public class RetryablePaymentProcessor {
    private static final int MAX_RETRIES = 3;
    private static final long INITIAL_DELAY = 1000;
    
    public PaymentResult processWithRetry(PaymentRequest request) {
        long delay = INITIAL_DELAY;
        
        for (int attempt = 1; attempt <= MAX_RETRIES; attempt++) {
            try {
                return process(request);
            } catch (TransientPaymentException e) {
                if (attempt == MAX_RETRIES) {
                    throw new PaymentException("Max retries exceeded", 
                        PaymentErrorCode.MAX_RETRIES_EXCEEDED);
                }
                try {
                    Thread.sleep(delay);
                    delay *= 2; // Exponential backoff
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    throw new PaymentException("Retry interrupted", 
                        PaymentErrorCode.RETRY_INTERRUPTED);
                }
            }
        }
        throw new PaymentException("Unexpected state", PaymentErrorCode.INTERNAL_ERROR);
    }
}

5. Testing & Test-Driven Design

  • Unit Testing: Test classes in isolation with mocks for dependencies
  • Test Doubles: Mock (verify behavior), Stub (provide data), Spy (partial mock)
  • BDD Structure: Given-When-Then for readable test cases
  • Coverage Goals: 80%+ coverage; focus on critical business logic
  • Integration Tests: Test actual service integrations beyond unit level

Unit Test with Mocks:

public class OrderServiceTest {
    private OrderService orderService;
    private PaymentProcessor paymentProcessor;
    private OrderRepository orderRepository;
    
    @Before
    public void setUp() {
        paymentProcessor = mock(PaymentProcessor.class);
        orderRepository = mock(OrderRepository.class);
        orderService = new OrderService(paymentProcessor, orderRepository);
    }
    
    @Test
    public void shouldCreateOrderWhenPaymentSucceeds() {
        // Given
        PaymentRequest request = new PaymentRequest("user123", BigDecimal.valueOf(100));
        when(paymentProcessor.process(request))
            .thenReturn(new PaymentResult(PaymentStatus.SUCCESS));
        
        // When
        Order order = orderService.createOrder(request);
        
        // Then
        assertNotNull(order);
        assertEquals(OrderStatus.CONFIRMED, order.getStatus());
        verify(orderRepository).save(order);
    }
    
    @Test
    public void shouldNotCreateOrderWhenPaymentFails() {
        // Given
        PaymentRequest request = new PaymentRequest("user123", BigDecimal.valueOf(100));
        when(paymentProcessor.process(request))
            .thenThrow(new PaymentException("Payment declined", 
                PaymentErrorCode.INSUFFICIENT_FUNDS));
        
        // When & Then
        assertThrows(PaymentException.class, () -> {
            orderService.createOrder(request);
        });
        verify(orderRepository, never()).save(any());
    }
}

High-Level Design (HLD)

Focuses on system architecture, component interactions, scalability, and reliability at macro level.

1. System Architecture Layers

  • Presentation Layer: User interfaces (Web, Mobile) - handles display and user input
  • API Gateway: Single entry point - routing, rate limiting, authentication, request translation
  • Business Logic Layer: Microservices - independent, deployable services with clear boundaries
  • Data Access Layer: Repositories, DAOs - abstract database interactions
  • Data Layer: Databases, caches, distributed stores - persistent data storage

2. Database Design & Sharding

  • Database Selection: SQL (ACID, transactions), NoSQL (scale, flexibility), NewSQL (both)
  • Vertical Sharding: Different entities in different databases (User DB, Order DB, Payment DB)
  • Horizontal Sharding: Same entity split by key (shard orders by user_id or date range)
  • Replication: Master-Slave (reads from replicas), Master-Master (active-active), or Raft consensus
  • Backup Strategy: Point-in-time recovery, incremental backups, geo-distributed redundancy

Sharding Implementation:

// Consistent Hash Sharding
public class ShardingRouter {
    private final TreeMap<Integer, String> ring = new TreeMap<>();
    private final int replicationFactor = 3;
    
    public ShardingRouter(List<String> shardNodes) {
        for (String node : shardNodes) {
            for (int i = 0; i < replicationFactor; i++) {
                int hash = hashFunction(node + "#" + i);
                ring.put(hash, node);
            }
        }
    }
    
    public String getShardForKey(String key) {
        if (ring.isEmpty()) return null;
        int hash = hashFunction(key);
        SortedMap<Integer, String> tailMap = ring.tailMap(hash);
        return tailMap.isEmpty() ? ring.firstEntry().getValue() : tailMap.get(tailMap.firstKey());
    }
    
    private int hashFunction(String key) {
        return key.hashCode();
    }
}

public class UserRepository {
    private final Map<String, Database> shards;
    private final ShardingRouter router;
    
    public void saveUser(User user) {
        String shardId = router.getShardForKey(user.getId());
        Database shard = shards.get(shardId);
        shard.insert("users", user);
    }
    
    public User getUser(String userId) {
        String shardId = router.getShardForKey(userId);
        return shards.get(shardId).query("users", userId);
    }
}

3. Caching Strategies

  • L1 Cache: In-application memory (HashMap, ConcurrentHashMap) - fastest, limited size
  • L2 Cache: Local Redis instance - fast, process-specific
  • L3 Cache: Distributed Redis cluster - shared across services
  • L4 Cache: CDN - for static content delivery
  • Cache Invalidation Patterns: TTL, Event-based, LRU/LFU eviction

Cache Patterns:

  • Cache-Aside: Check cache, miss → fetch DB → update cache (most common)
  • Write-Through: Write to cache AND DB together (consistent but slower)
  • Write-Behind: Write to cache immediately, async update DB (fast but risky)
  • Refresh-Ahead: Proactively refresh before TTL expires

Cache-Aside Pattern:

public class UserServiceWithCache {
    private final UserRepository userRepository;
    private final RedisCache cache;
    private static final String CACHE_KEY_PREFIX = "user:";
    private static final long CACHE_TTL_SECONDS = 3600;
    
    public User getUser(String userId) {
        // Step 1: Check cache
        String cacheKey = CACHE_KEY_PREFIX + userId;
        User cachedUser = cache.get(cacheKey, User.class);
        if (cachedUser != null) return cachedUser;
        
        // Step 2: Cache miss - fetch from DB
        User user = userRepository.findById(userId);
        
        // Step 3: Update cache
        if (user != null) {
            cache.set(cacheKey, user, CACHE_TTL_SECONDS);
        }
        return user;
    }
    
    public void updateUser(User user) {
        userRepository.update(user);
        
        // Invalidate cache
        cache.delete(CACHE_KEY_PREFIX + user.getId());
    }
    
    // Warm cache for performance
    public void warmCache(List<String> userIds) {
        for (String userId : userIds) {
            if (!cache.exists(CACHE_KEY_PREFIX + userId)) {
                User user = userRepository.findById(userId);
                if (user != null) {
                    cache.set(CACHE_KEY_PREFIX + userId, user, CACHE_TTL_SECONDS);
                }
            }
        }
    }
}

4. Message Queuing & Async Processing

  • Event-Driven Architecture: Decouple producers and consumers using message queues
  • Asynchronous Processing: Non-blocking request handling via background workers
  • Event Sourcing: Store all changes as immutable events; replay to rebuild state
  • Message Ordering: Ensure messages for same entity processed in order (partition by key)
  • Idempotency: Design handlers to safely handle duplicate messages
  • Dead Letter Queue: Route failed messages for manual inspection

Event Processing with Idempotency:

public interface OrderEvent {
    String getOrderId();
    LocalDateTime getTimestamp();
}

public class OrderCreatedEvent implements OrderEvent {
    private final String orderId;
    private final String userId;
    private final BigDecimal amount;
    private final LocalDateTime timestamp;
    
    public OrderCreatedEvent(String orderId, String userId, BigDecimal amount) {
        this.orderId = orderId;
        this.userId = userId;
        this.amount = amount;
        this.timestamp = LocalDateTime.now();
    }
    
    @Override
    public String getOrderId() { return orderId; }
    @Override
    public LocalDateTime getTimestamp() { return timestamp; }
}

public class PaymentEventConsumer {
    private final PaymentService paymentService;
    private final ProcessedEventStore processedEvents;
    
    public void handleOrderCreatedEvent(OrderCreatedEvent event) {
        // Idempotency: Check if already processed
        if (processedEvents.exists(event.getOrderId())) {
            return; // Skip duplicate
        }
        
        try {
            paymentService.processPayment(
                event.getOrderId(),
                event.getUserId(),
                event.getAmount()
            );
            
            // Mark as processed
            processedEvents.mark(event.getOrderId());
        } catch (Exception e) {
            // Publish to DLQ for manual inspection
            throw new ProcessingException("Payment failed", e);
        }
    }
}

SOLID Principles

Five design principles for maintainable, flexible, and scalable code.

1. Single Responsibility Principle (SRP)

A class should have only one reason to change.

  • Each class has ONE well-defined responsibility
  • High cohesion within the class
  • Easier to test, reuse, and maintain
  • Avoid "God Classes" that do too many things
  • When requirements change, only related classes need updates

Bad Example - Violates SRP:

// BAD: This class has multiple responsibilities
public class UserService {
    public void createUser(User user) { /* database operation */ }
    public void sendWelcomeEmail(User user) { /* email operation */ }
    public void logUserAction(String userId, String action) { /* logging */ }
    public boolean validateUser(User user) { /* validation */ }
}

Good Example - Follows SRP:

// GOOD: Separate concerns into different classes
public class UserService {
    private final UserRepository repository;
    private final EmailService emailService;
    private final UserValidator validator;
    
    public void createUser(User user) {
        if (validator.isValid(user)) {
            repository.save(user);
            emailService.sendWelcomeEmail(user);
        }
    }
}

public class UserRepository {
    public void save(User user) { }
}

public class EmailService {
    public void sendWelcomeEmail(User user) { }
}

public class UserValidator {
    public boolean isValid(User user) { }
}

2. Open/Closed Principle (OCP)

Classes should be open for extension but closed for modification.

  • Design classes so new features can be added without modifying existing code
  • Use abstract classes, interfaces, and inheritance for extension
  • Reduces risk of breaking existing functionality
  • Enables plugin-based architectures
  • Use Strategy, Template Method, or Decorator patterns

Bad Example - Violates OCP:

// BAD: Adding new payment types requires modification
public class PaymentProcessor {
    public void processPayment(String type, PaymentDetails details) {
        if ("CREDIT_CARD".equals(type)) {
            processCreditCard(details);
        } else if ("UPI".equals(type)) {
            processUPI(details);
        }
        // Need to modify this class every time!
    }
}

Good Example - Follows OCP:

// GOOD: Closed for modification, open for extension
public interface PaymentGateway {
    PaymentResult process(PaymentDetails details);
}

public class CreditCardGateway implements PaymentGateway {
    @Override
    public PaymentResult process(PaymentDetails details) { }
}

public class UPIGateway implements PaymentGateway {
    @Override
    public PaymentResult process(PaymentDetails details) { }
}

public class PaymentProcessor {
    private final Map<String, PaymentGateway> gateways;
    
    public PaymentResult processPayment(String type, PaymentDetails details) {
        return gateways.get(type).process(details);
    }
    
    // Register new payment types without modifying this class!
    public void registerGateway(String type, PaymentGateway gateway) {
        gateways.put(type, gateway);
    }
}

3. Liskov Substitution Principle (LSP)

Derived classes must be substitutable for their base classes.

  • If B extends A, we should be able to replace A with B without breaking the program
  • Don't weaken preconditions or strengthen postconditions in subclasses
  • Maintain the contract defined by the parent class
  • Avoid behavioral changes in subclasses
  • Use composition over inheritance when inheritance doesn't satisfy LSP

Bad Example - Violates LSP:

// BAD: Square violates Rectangle contract
public class Rectangle {
    protected int width, height;
    public void setWidth(int w) { this.width = w; }
    public void setHeight(int h) { this.height = h; }
    public int getArea() { return width * height; }
}

public class Square extends Rectangle {
    @Override
    public void setWidth(int w) {
        this.width = w;
        this.height = w; // Violates contract!
    }
}

// This breaks with Square!
void testRectangle(Rectangle rect) {
    rect.setWidth(5);
    rect.setHeight(4);
    assert rect.getArea() == 20; // FAILS with Square!
}

Good Example - Follows LSP:

// GOOD: Don't force inheritance where it doesn't fit
public abstract class Shape {
    public abstract int getArea();
}

public class Rectangle extends Shape {
    private int width, height;
    public Rectangle(int w, int h) { this.width = w; this.height = h; }
    public void setWidth(int w) { this.width = w; }
    public void setHeight(int h) { this.height = h; }
    @Override
    public int getArea() { return width * height; }
}

public class Square extends Shape {
    private int side;
    public Square(int side) { this.side = side; }
    public void setSide(int s) { this.side = s; }
    @Override
    public int getArea() { return side * side; }
}

4. Interface Segregation Principle (ISP)

Clients should not depend on interfaces they don't use.

  • Create specific, focused interfaces rather than large all-purpose ones
  • Reduce coupling by depending only on needed methods
  • Easier to mock for testing with small interfaces
  • Prevents "fat interfaces" forcing unwanted implementations
  • Multiple inheritance (multiple interfaces) more flexible than single base class

Bad Example - Violates ISP:

// BAD: Fat interface - Robot must implement eat() and sleep()
public interface Worker {
    void work();
    void eat();
    void sleep();
}

public class Robot implements Worker {
    public void work() { }
    public void eat() { throw new UnsupportedOperationException(); }
    public void sleep() { throw new UnsupportedOperationException(); }
}

Good Example - Follows ISP:

// GOOD: Segregate into focused interfaces
public interface Workable {
    void work();
}

public interface Eatable {
    void eat();
}

public interface Sleepable {
    void sleep();
}

public class Human implements Workable, Eatable, Sleepable { }
public class Robot implements Workable { }

public class WorkManager {
    public void manageWorkers(List<Workable> workers) {
        for (Workable w : workers) w.work();
    }
}

5. Dependency Inversion Principle (DIP)

High-level modules should not depend on low-level modules. Both should depend on abstractions.

  • Depend on interfaces/abstractions, not concrete implementations
  • High-level code shouldn't know implementation details
  • Enables flexibility - swap implementations easily
  • Supports testing with mock implementations
  • Use dependency injection to provide dependencies

Bad Example - Violates DIP:

// BAD: Tight coupling to concrete class
public class OrderService {
    private MySQLDatabase database = new MySQLDatabase();
    
    public void createOrder(Order order) {
        database.save(order.toString());
    }
}
// Problem: Can't switch to MongoDB or PostgreSQL

Good Example - Follows DIP:

// GOOD: Depend on abstraction
public interface Database {
    void save(String data);
}

public class MySQLDatabase implements Database {
    public void save(String data) { }
}

public class OrderService {
    private final Database database; // Interface, not concrete
    
    public OrderService(Database database) {
        this.database = database;
    }
    
    public void createOrder(Order order) {
        database.save(order.toString());
    }
}

// Easy to swap implementations
Database db = new MySQLDatabase();
OrderService service = new OrderService(db);

CAP Theorem

A distributed system can guarantee at most TWO of three properties: Consistency, Availability, Partition Tolerance.

Understanding CAP Properties

  • Consistency (C): Every read returns the most recent write; all nodes see same data
  • Availability (A): Every request gets response; system remains operational even if nodes down
  • Partition Tolerance (P): System continues operation despite network partitions
  • The Trade-off: During network partition, MUST choose between C and A
  • 2026 Perspective: NewSQL databases achieve "2.5" properties using eventual consistency

1. CA Systems - Consistency + Availability

  • Trade-off: Cannot tolerate network partitions; requires all nodes connected
  • Best for: Single datacenter, internal systems, ACID transactions critical
  • Examples: PostgreSQL, MySQL, Oracle (single region)
  • Approach: Synchronous replication, fail if partition occurs
  • Modern use: Stretch cluster across zones with low latency

CA Implementation - PostgreSQL:

// CA System - Synchronous Replication (Fast, Consistent)
public class ConsistentOrderService {
    private final JdbcTemplate jdbcTemplate;
    
    public void createOrder(Order order) {
        // Synchronous replication - waits for replica acknowledgment
        try {
            jdbcTemplate.update(
                "INSERT INTO orders (id, user_id, amount) VALUES (?, ?, ?)",
                order.getId(),
                order.getUserId(),
                order.getAmount()
            );
            // Returns only after all replicas acknowledged
        } catch (DataAccessException e) {
            // Partition or replica failure
            throw new OrderProcessingException("Failed to create order", e);
        }
    }
    
    public Order getOrder(String orderId) {
        // Always reads from primary - guarantees latest data
        return jdbcTemplate.queryForObject(
            "SELECT * FROM orders WHERE id = ?",
            new OrderRowMapper(),
            orderId
        );
    }
}

2. CP Systems - Consistency + Partition Tolerance

  • Trade-off: Sacrifices Availability; blocks requests during partition
  • Best for: Financial systems, inventory, correctness critical
  • Examples: MongoDB (majority writes), Redis Cluster, HBase
  • Approach: Quorum writes, consensus algorithms (Raft, Paxos)
  • Pattern: Requires majority acknowledgment for writes

CP Implementation - MongoDB:

// CP System - Majority Write Concern (Consistent across partitions)
public class ConsistentInventoryService {
    private final MongoTemplate mongoTemplate;
    
    public void decrementStock(String productId, int quantity) {
        // Majority write concern - requires acknowledgment from majority
        WriteConcern writeConcern = WriteConcern.MAJORITY.withWTimeout(5, TimeUnit.SECONDS);
        
        try {
            Query query = Query.query(Criteria.where("productId").is(productId));
            Update update = new Update().inc("quantity", -quantity);
            
            mongoTemplate.updateFirst(query, update, Product.class);
            
            // Only returns after majority acknowledged - survives minority failures
        } catch (MongoWriteConcernException e) {
            // Partition detected - cannot achieve majority
            throw new PartitionException("Cannot process write during partition", e);
        }
    }
    
    public int getStock(String productId) {
        // Read from primary only - may fail if primary isolated
        Product product = mongoTemplate.findById(productId, Product.class);
        return product != null ? product.getQuantity() : 0;
    }
}

4. PACELC Theorem - Real-World Trade-offs (2026+)

  • PACELC: If (P)artition, choose (A)vailability or (C)onsistency; Else choose (L)atency or (C)onsistency
  • During Normal Operation: Trade-off between latency and consistency
  • Hybrid Approaches: Different components handle differently (orders CP, recommendations AP)
  • Strong Eventual Consistency: Read-your-write consistency, causal consistency guarantees
  • NewSQL Solutions: CockroachDB, Spanner, Vitess - try to get "2.5" properties

Hybrid CAP Strategy:

// PACELC - Choose between latency and consistency based on use case
public class HybridOrderService {
    private final ConsistentDatabase consistentDB; // PostgreSQL - CP
    private final AvailableCache availableCache;   // Redis - AP
    
    // For critical operations - prioritize consistency over latency
    public void createOrder(Order order) {
        // CP: Consistent and partition-tolerant
        consistentDB.save(order);
    }
    
    // For reading - trade consistency for latency
    public List<Order> getUserOrders(String userId) {
        // Try cache first (available, potentially stale)
        List<Order> cached = availableCache.getOrders(userId);
        if (cached != null) return cached;
        
        // Cache miss - read from consistent source
        List<Order> orders = consistentDB.getOrders(userId);
        availableCache.setOrders(userId, orders, 3600);
        
        return orders;
    }
}

Design Patterns (Creational, Structural, Behavioral)

Reusable solutions to common design problems in software development.

Creational Patterns - Object Creation

1. Singleton Pattern

  • Ensures ONE instance of a class throughout application lifetime
  • Use cases: Database connection pool, Logger, Configuration manager
  • Thread-safe: Bill Pugh Singleton pattern (class loader guarantees thread safety)
  • Downside: Hides dependencies, makes testing harder
  • Modern alternative: Dependency injection (Spring @Singleton)

2. Factory Pattern

  • Creates objects without specifying exact classes
  • Use cases: PaymentGatewayFactory creates different payment processors
  • Encapsulates object creation logic
  • Simple Factory vs Factory Method vs Abstract Factory

3. Builder Pattern

  • Constructs complex objects step-by-step
  • Best for objects with many optional parameters
  • Improves readability of construction code
  • Fluent interface for chaining method calls

Structural Patterns - Object Composition

1. Adapter Pattern

  • Converts interface of a class to another interface clients expect
  • Use cases: Integrating legacy systems, third-party APIs with different interfaces
  • Example: StripePaymentAdapter makes Stripe API compatible with PaymentGateway interface

2. Decorator Pattern

  • Adds new functionality to object dynamically without modifying original
  • Use cases: Adding logging, caching, validation to methods
  • More flexible than inheritance
  • Example: CachedPaymentGateway wraps PaymentGateway with caching logic

3. Facade Pattern

  • Provides simplified interface to complex subsystem
  • Use cases: Order processing coordinating payment, inventory, notification
  • Reduces coupling between client and subsystem
  • Single entry point to multiple services

Behavioral Patterns - Object Interaction

1. Observer Pattern

  • One-to-many dependency: when one changes, all dependents notified
  • Use cases: Event systems, pub-sub messaging, MVC pattern
  • Example: Order status changes notify EmailService, SMSService, LogService

2. Strategy Pattern

  • Family of algorithms, encapsulates each one, makes them interchangeable
  • Use cases: Different discount strategies, payment methods, sorting algorithms
  • Runtime algorithm selection based on context

3. State Pattern

  • Object alters behavior when internal state changes
  • Use cases: Order workflow (Pending → Confirmed → Shipped → Delivered)
  • Each state handles its own transitions and behaviors

4. Template Method Pattern

  • Define algorithm skeleton in base class, subclasses implement specific steps
  • Use cases: Different payment processing flows with common steps
  • Prevents code duplication of common workflow

Architectural Patterns (2026+)

1. CQRS (Command Query Responsibility Segregation)

  • Separate read and write models for optimized scaling
  • Commands modify state, Queries retrieve state
  • Write model optimized for transactions, Read model for performance
  • Use cases: Complex domains with different read/write patterns

2. Event Sourcing

  • Store all state changes as immutable events, rebuild state by replaying
  • Complete audit trail of all changes
  • Can replay events to any point in time
  • Use cases: Financial systems, audit requirements, temporal queries

3. Saga Pattern

  • Manage long-running, multi-step transactions in microservices
  • Choreography (event-driven) vs Orchestration (central coordinator)
  • Implements compensating transactions for failure handling
  • Use cases: Order processing across multiple services

4. Circuit Breaker

  • Prevent cascading failures by monitoring and failing fast
  • States: Closed (normal), Open (failing), Half-Open (testing recovery)
  • Use cases: External API calls, microservice communication

C. Matching Engine - Driver Assignment Algorithm

// Matching Engine - Sophisticated driver selection
public class MatchingEngine {
    private final LocationService locationService;
    private final DriverRepository driverRepository;
    private final RideRepository rideRepository;
    private final EventPublisher eventPublisher;
    private final RideCostCalculator costCalculator;
    
    public void matchRide(RideRequest request) {
        // Step 1: Get nearby drivers
        List<String> nearbyDriverIds = locationService.findNearbyDrivers(
            request.getPickupLat(),
            request.getPickupLng(),
            10
        );
        
        // Step 2: Score drivers based on multiple factors
        List<DriverScore> scoredDrivers = nearbyDriverIds.stream()
            .map(id -> scoreDriver(id, request))
            .sorted(Comparator.comparingDouble(DriverScore::getScore).reversed())
            .limit(5)
            .collect(Collectors.toList());
        
        // Step 3: Try to match with top drivers
        for (DriverScore score : scoredDrivers) {
            if (sendMatchRequest(score.getDriverId(), request)) {
                createRide(score.getDriverId(), request);
                return;
            }
        }
        
        // No drivers available - queue the request
        rideRepository.addToQueue(request);
        eventPublisher.publish("ride-queued", new RideQueuedEvent(request));
    }
    
    private DriverScore scoreDriver(String driverId, RideRequest request) {
        Driver driver = driverRepository.findById(driverId).orElseThrow();
        
        double score = 0;
        
        // Distance (0.4 weight)
        double distance = calculateDistance(
            driver.getLatitude(),
            driver.getLongitude(),
            request.getPickupLat(),
            request.getPickupLng()
        );
        score += (100 - distance * 2) * 0.4;
        
        // Rating (0.3 weight)
        score += (driver.getRating() / 5.0 * 100) * 0.3;
        
        // Acceptance rate (0.2 weight)
        score += driver.getAcceptanceRate() * 100 * 0.2;
        
        // Current load (0.1 weight)
        score += (100 - driver.getActiveRides() * 10) * 0.1;
        
        return new DriverScore(driverId, score);
    }
    
    private boolean sendMatchRequest(String driverId, RideRequest request) {
        try {
            DriverResponse response = websocketManager.waitForResponse(
                driverId,
                request.getId(),
                5 // 5 second timeout
            );
            return response.isAccepted();
        } catch (TimeoutException e) {
            return false;
        }
    }
    
    private void createRide(String driverId, RideRequest request) {
        Ride ride = new Ride(
            UUID.randomUUID().toString(),
            request.getRiderId(),
            driverId,
            request.getPickupLocation(),
            request.getDropoffLocation(),
            RideStatus.ASSIGNED
        );
        
        // Calculate cost
        BigDecimal cost = costCalculator.calculateCost(
            request.getPickupLocation(),
            request.getDropoffLocation(),
            request.getRideType()
        );
        ride.setCost(cost);
        
        rideRepository.save(ride);
        eventPublisher.publish("ride-created", new RideCreatedEvent(ride));
    }
}

// Payment Service with Strategy
public class PaymentService {
    private final Map<String, PaymentGateway> gateways;
    private final PaymentRepository repository;
    private final EventPublisher eventPublisher;
    private final RetryStrategy retryStrategy;
    
    public PaymentResult processPayment(PaymentRequest request) {
        Payment payment = new Payment(
            request.getRideId(),
            request.getUserId(),
            request.getAmount(),
            request.getPaymentMethod()
        );
        
        repository.save(payment);
        
        try {
            PaymentGateway gateway = gateways.get(request.getPaymentMethod().name());
            PaymentResult result = retryStrategy.executeWithRetry(() ->
                gateway.process(request)
            );
            
            if (result.isSuccess()) {
                payment.setStatus(PaymentStatus.SUCCESS);
                repository.update(payment);
                eventPublisher.publish("payment-success",
                    new PaymentSuccessEvent(payment, result.getTransactionId())
                );
            } else {
                payment.setStatus(PaymentStatus.FAILED);
                repository.update(payment);
                eventPublisher.publish("payment-failed",
                    new PaymentFailedEvent(payment, result.getError())
                );
            }
            
            return result;
        } catch (Exception e) {
            payment.setStatus(PaymentStatus.FAILED);
            repository.update(payment);
            throw new PaymentProcessingException("Payment failed", e);
        }
    }
}

Scalability & Resilience Strategies

  • Sharding: Shard users/rides by user_id using consistent hashing for horizontal scaling
  • Caching: Redis cluster for location cache, user sessions, geohash index
  • Message Queue: Kafka for asynchronous processing (payment, notifications, analytics)
  • Circuit Breaker: Protect against payment gateway failures with automatic fallback
  • Load Balancing: Round-robin with health checks, sticky sessions for WebSocket connections
  • Replication: Multi-region deployment for disaster recovery and low-latency access
  • Monitoring: Distributed tracing, structured logging, metrics collection with alerting
